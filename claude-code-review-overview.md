# Claude Code コードレビュー機能まとめ

Claudeのコードレビュー機能は **3種類** ある（マネージド・Code Review・PR Review Toolkit）。
それぞれ目的・仕組み・コストが異なるので、用途に合わせて選ぶ必要がある。

> Superpowersにもレビュー関連のスキルがあるが、これらは「コードレビュー」ではなくセルフチェック・指摘対応のプロセスなので、本ページでは補足として扱う。

---

## 「マネージド」って何？

「マネージド（Managed）」＝ **Anthropic側のサーバーで自動実行される仕組み** のこと。
ユーザーが手動でコマンドを打つ必要がなく、PRが作られると勝手にレビューが走る。
逆に「プラグイン/スキル」は **ローカルのClaude Codeセッション内で自分で起動する** もの。

| | マネージド | プラグイン/スキル |
|---|---|---|
| 実行場所 | Anthropicのサーバー | 自分のPC（ローカル） |
| 起動方法 | PRを開くと自動 | コマンドで手動起動 |
| 必要なもの | Teams/Enterpriseプラン | Claude Codeがあればok |
| コスト | 高い（$15〜25/PR） | 自分のトークン消費分 |

---

## 1. マネージド Code Review（Anthropic公式・サーバーサイド）

**概要**: PRが開かれると、Anthropicのサーバー上で複数エージェントが自動でレビューを実行する。

**仕組み**:
- PRが作成されると自動でエージェントチームが割り当てられる
- エージェントが**並列で**バグ探索 → 検証（誤検知除外）→ 重大度ランク付け
- 結果はGitHubにインラインコメントとして投稿
- 大規模PRほどエージェント数が増える（スケーリング）

**検出実績（Anthropic社内データ）**:
- 1,000行以上のPR → **84%でバグ発見**、平均7.5件
- 50行未満のPR → 31%で発見、平均0.5件

**料金**: PR1件あたり平均 **$15〜25**（トークン使用量ベース）
**対象**: Teams / Enterprise のみ

---

## 2. Code Review プラグイン（Anthropic公式・ローカル実行）

**概要**: マネージド版のローカル版。`/code-review` コマンドで手動起動する。

**仕組み**:
- **4つの並列エージェント**を起動:
  - エージェント #1・#2: CLAUDE.md準拠チェック（プロジェクトのルールに違反してないか）
  - エージェント #3: 変更差分からの明らかなバグスキャン
  - エージェント #4: git blame/履歴を分析して文脈ベースの問題を検出
- 各発見に**0〜100の信頼度スコア**をつける
- **80点未満は自動フィルタ**（誤検知を大幅に減らす）
- 既存の問題、些末な指摘、lintで捕まるものは自動スキップ
- クローズ済み・ドラフト・些末なPRもスキップ

**マネージド版との違い**:
- 自分のタイミングで起動できる
- 特定のPRだけ選んでレビューできる
- コスト制御しやすい（自分のトークン消費のみ）

---

## 3. PR Review Toolkit プラグイン（Anthropic公式・ローカル実行）

**概要**: Code Reviewプラグインの**重量版**。6つの専門エージェントでより多角的にレビューする。

**6つの専門エージェント**:

| エージェント | 役割 |
|---|---|
| **code-reviewer** | コーディング規約・バグの一般レビュー（信頼度80以上のみ報告） |
| **comment-analyzer** | コードコメントの正確性・メンテナンス性チェック |
| **pr-test-analyzer** | テストカバレッジの質と漏れを1〜10でスコアリング |
| **silent-failure-hunter** | エラーハンドリングの欠陥、サイレントに失敗する箇所の検出 |
| **type-design-analyzer** | 型設計の品質を4軸で1〜10評価 |
| **code-simplifier** | 不要な複雑さの特定、リファクタリング提案 |

**Code Reviewプラグイン（軽量版）との関係**:
- Code Reviewは4エージェント・軽量・高速 → **日常のPR向き**
- PR Review Toolkitは6エージェント・重量・多角的 → **重要なPR・リリース前向き**
- 上位互換ではなく**「軽量版と重量版」**の関係
- 全PRにToolkitを回すのはトークンの無駄遣い。使い分けが前提の設計

## （補足）Superpowers: requesting-code-review

> コードレビューではなく、**開発中のセルフチェック**。作りながら「ここまで大丈夫？」を都度確認する仕組み。

**概要**: 開発フローの中に組み込まれるセルフチェックスキル。タスク完了時に **サブエージェント（`superpowers:code-reviewer`）を派遣** して品質を確認させる。

**仕組み**:
- git SHAの範囲を指定して、変更差分をレビューエージェントに渡す
- レビューエージェントが以下をチェック:
  - コード品質、アーキテクチャ、テスト、要件充足、本番投入可否
- 結果を **Critical / Important / Minor** に分類
- **Criticalは修正するまでブロック**（マージさせない）
- 「マージ可能か？」を技術的根拠付きで判定

**発動タイミング**:
- 必須: サブエージェント開発の各タスク後、大きな機能実装後、mainマージ前
- 任意: 行き詰まった時、リファクタ前、複雑なバグ修正後

**他との違い**:
- PRベースではなく**開発中にこまめにレビュー**する思想
- 「レビューは早く、頻繁に（Review early, review often）」が原則
- TDD → 実装 → レビュー → 完了 という開発フロー全体に統合

---

## （補足）Superpowers: receiving-code-review

> コードレビューではなく、**レビュー指摘への対応プロセス**。指摘をどう扱うかのスキル。

**概要**: レビューを**受ける側**のスキル。フィードバックに対して盲目的に従わず、技術的に検証してから対応する。

**仕組み（6ステップ）**:
1. **READ** — フィードバックを読む
2. **UNDERSTAND** — 意図を理解する
3. **VERIFY** — 技術的に正しいか検証する
4. **EVALUATE** — 実装すべきか判断する
5. **RESPOND** — 根拠をもって回答する
6. **IMPLEMENT** — 必要なものだけ実装する

**禁止されている対応**:
- 「おっしゃる通りです！」「素晴らしい指摘！」→ パフォーマティブな同意は禁止
- 指摘されたらまず直す、ではなく **まず検証**

**プッシュバック（反論）すべき場面**:
- 機能を壊す修正
- コンテキスト不足の指摘
- YAGNI違反（使われてない機能の追加要求）
- 技術的に不正確
- アーキテクチャ方針と矛盾

**他との違い**:
- 唯一の**「受け身側」のスキル**
- 人間のレビューにもAIのレビューにも適用される
- 「指摘 = 正しい」ではなく「指摘 = 検証が必要」という姿勢

---

## （番外）verification-before-completion

レビューそのものではないが、関連が深いスキル。

- **「完了」を宣言する前に必ず証拠を出せ**
- テスト通った？ → 実際にテスト走らせてログを見せろ
- ビルド通った？ → 実際にビルドコマンド走らせてログを見せろ
- 「たぶん大丈夫」「自信ある」は無効
- Superpowersの品質保証の最終ゲート

## 3つの比較

| | マネージド | Code Review プラグイン | PR Review Toolkit |
|---|---|---|---|
| 提供元 | Anthropic | Anthropic | Anthropic |
| 実行場所 | サーバー | ローカル | ローカル |
| 起動 | PR作成で自動 | `/code-review` | `/review-pr` |
| エージェント数 | 可変（PR規模次第） | 4 | 6 |
| 観点 | バグ・セキュリティ | CLAUDE.md準拠・バグ・履歴 | バグ・テスト・型・コメント・複雑度・エラー処理 |
| 対象プラン | Teams/Enterprise | 全プラン | 全プラン |
| コスト | $15〜25/PR | トークン分 | トークン分（多め） |
| 向いてる場面 | 全PRを自動レビュー | 日常のPR | 重要なPR・リリース前 |

---

## なぜ3種類あるのか？

| 理由 | 説明 |
|---|---|
| 実行形態が違う | サーバー自動 vs ローカル手動 |
| 重さが違う | 軽量4エージェント vs 重量6エージェント |
| コストが違う | $15〜25/PR vs 自分のトークン分 |
| 対象プランが違う | Teams/Enterprise限定 vs 全プラン |

---

## どれが優れているか？

| 観点 | 最も優れている | 理由 |
|---|---|---|
| バグ発見能力 | **マネージド** | 1,000行以上PRで84%発見率・平均7.5件は圧倒的 |
| コスパ | **Code Reviewプラグイン** | マネージドと同系統の仕組みを自分のトークンで動かせる |
| レビューの深さ | **PR Review Toolkit** | 6専門エージェントで最も多角的（テスト・型・コメント・エラー処理まで） |

万能なツールはない。場面で使い分ける。

---

## どう使い分ける？

- **個人開発** → Code Reviewプラグイン1つで十分
- **チーム開発（予算あり）** → マネージド1つで十分
- **日常PR** → Code Reviewプラグイン（軽量・高速）
- **重要PR・リリース前** → PR Review Toolkit（多角的・徹底）
