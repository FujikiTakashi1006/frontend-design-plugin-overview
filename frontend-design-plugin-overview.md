# Claude Code「Frontend Design」プラグイン概要

---

## 1. Frontend Designプラグインとは？

- Claude Codeに追加できる**フロントエンドデザイン特化のプラグイン**
- Anthropic公式が開発・公開
- 「AIっぽい量産型デザイン（AIスロップ）」を避け、**個性的で高品質なUI**を生成する

---

## 2. 何が変わるのか

### Before（プラグインなし）

- ありきたりなフォント（Inter, Arial, Roboto）
- 紫グラデーション on 白背景のような定番配色
- 予測可能なレイアウト
- どのAIで作っても同じに見える

### After（プラグインあり）

- **大胆なフォント選び**（個性的なディスプレイフォント × 洗練されたボディフォント）
- **コンセプトに基づく配色**（ブルータリスト、レトロフューチャー、ラグジュアリーなど）
- **非対称・グリッド崩し**の印象的なレイアウト
- **アニメーション・マイクロインタラクション**による演出
- **テクスチャ・グラデーションメッシュ・ノイズ**などの背景効果

## 3. 仕組み

### プラグインの構造（たった4ファイル）

```
frontend-design/
├── .claude-plugin/
│   └── plugin.json       ← マニフェスト（名前・説明・著者）
├── skills/
│   └── frontend-design/
│       └── SKILL.md      ← 本体（デザイン指示プロンプト）
├── README.md
└── LICENSE
```

### 動作の流れ

1. **マーケットプレイス**からインストール（GitHubリポジトリベース）
2. `settings.json` の `enabledPlugins` で有効化
3. フロントエンド制作の依頼を検知
4. **SKILL.mdの内容がClaudeのコンテキストに注入**される
5. デザイン指示に従い、個性的なUIコードを出力

### 核心

- プログラム的な拡張ではない
- **「条件付きで追加されるシステムプロンプト」**
- プロンプトエンジニアリングによるデザイン品質の制御

### 注入されるプロンプトの中身（SKILL.md：全42行）

プロンプトは大きく3パートで構成：

#### Part 1: Design Thinking — 「コードの前にまず考えろ」

- **Purpose** — 誰のため？何を解決する？
- **Tone** — 美的方向性を「極端に」選べ（ブルータリスト、レトロフューチャー、ラグジュアリー等）
- **Constraints** — 技術的制約の確認
- **Differentiation** — 何が「忘れられない」ポイントか？

#### Part 2: Frontend Aesthetics Guidelines — 5つの具体的指示

| 領域                | 指示内容                                             |
| ------------------- | ---------------------------------------------------- |
| Typography          | ありきたりなフォント禁止、個性的なフォントペアリング |
| Color & Theme       | CSS変数で一貫性、支配色＋鋭いアクセント              |
| Motion              | アニメーション重視、ページロード時のスタガー演出     |
| Spatial Composition | 非対称、オーバーラップ、グリッド崩し                 |
| Backgrounds         | テクスチャ、ノイズ、グラデーションメッシュで奥行き   |

#### Part 3: 禁止リスト — 「AIスロップ」の明示的な定義

- Inter, Roboto, Arial, system fonts → 禁止
- 紫グラデーション on 白背景 → 禁止
- 予測可能なレイアウト → 禁止
- Space Grotesk → 名指しで「毎回これに収束するな」と禁止

**ポイント：技術指示ではなく「美意識の注入」**

- 「Reactを使え」「Tailwindを使え」のような技術指定は一切ない
- 「大胆であれ」「毎回違うデザインにしろ」「AIっぽいのは禁止」という**姿勢・価値観の指示**

### なぜこの42行でうまくいくのか？（プロンプト設計の分析）

> ※ 以下は開発元の公式見解ではなく、SKILL.mdの構造から読み取れる設計意図の分析です

#### 1.「何をしろ」ではなく「どう考えろ」

- 「Google Fontsから選べ」「Tailwindを使え」のような具体的手順は一切ない
- 代わりに「大胆な美的方向性にコミットしろ」「忘れられないものを作れ」という**思考フレームワーク**を与えている
- Claudeの推論能力を信頼し、「やり方」を縛らず「考え方」だけ指定 → 応用が効く

#### 2.「やれ」より「やるな」が効く

- 最もインパクトがあるのは禁止リスト（NEVER指示）
- LLMは学習データの**最頻出パターンに収束しやすい**（放っておくとInter + 紫グラデ + 整列レイアウトになる）
- 「いいデザインをしろ」だけでは収束を防げないので、**収束先を明示的にブロック**している

#### 3.「毎回違え」— 再現性の意図的な破壊

- `No design should be the same` という指示
- 同じプロンプトで同じ出力が出ると「AIっぽさ」が際立つ
- **再現性そのものを否定**して、毎回違う方向に振らせる戦略

#### 4. 逆三角形の構造

```
抽象（Design Thinking: 考え方・姿勢）
  ↓
具体（Aesthetics Guidelines: 5領域の指針）
  ↓
制約（禁止リスト: やるな）
```

- まず広く考えさせて → 具体的な観点を与えて → NGゾーンで絞る
- **自由度を保ちつつ品質の下限を保証**する構造

#### 5. 最後の1行が「リミッター解除」

- `Don't hold back, show what can truly be created`（遠慮するな、本当に作れるものを見せろ）
- LLMは安全側・無難側に倒れやすい → この1行で**リミッターを解除**している

#### 設計原則まとめ

| 設計原則            | 実現方法                               |
| ------------------- | -------------------------------------- |
| LLMの収束傾向を防ぐ | 禁止リストで最頻出パターンをブロック   |
| 毎回違う出力にする  | 再現性そのものを否定する指示           |
| 応用力を保つ        | 手順ではなく思考フレームワークを与える |
| 品質の下限を保証    | 抽象→具体→制約の逆三角形構造           |
| 無難に逃げさせない  | 最後の1行でリミッター解除              |

## 4. プラグインとコンテキストの関係

### コンテキスト消費量

|                                | トークン数             |
| ------------------------------ | ---------------------- |
| SKILL.md全文                   | **約500〜600トークン** |
| Claudeのコンテキストウィンドウ | **1,000,000トークン**  |
| 占有率                         | **約0.05%**            |

- コンテキストを**ほぼ荒らさない**設計
- **費用対効果は最もコスパの良いプラグインの一つ**

### スキル発動時のコンテキスト注入の仕組み

Claude Codeのプラグイン（スキル）は、以下の流れでコンテキストに載る：

```
ユーザーの依頼
  ↓
Claude Codeがスキルの発動条件に合致するか判定
  ↓ （合致した場合）
該当スキルのSKILL.mdをコンテキストに注入
  ↓
Claudeがスキルの指示に従って応答を生成
```

- スキルは**常に載っているわけではない** — 発動条件にマッチしたときだけ注入される
- 注入されたプロンプトは、そのターンのシステムプロンプトの一部として扱われる
- つまり、使わないスキルが勝手にコンテキストを消費することはない

### 複数プラグインを併用した場合の影響

実際に本環境にインストールされているプラグインで比較：

#### Frontend Design（1スキル）

|                 | トークン数 |
| --------------- | ---------- |
| frontend-design | **~550**   |
| **合計**        | **~550**   |

→ 発動しても**0.05%**しかコンテキストを使わない

#### Superpowers（14スキル）

| スキル名                       | トークン数  |
| ------------------------------ | ----------- |
| writing-skills                 | ~5,616      |
| subagent-driven-development    | ~3,034      |
| brainstorming                  | ~2,642      |
| systematic-debugging           | ~2,471      |
| test-driven-development        | ~2,466      |
| dispatching-parallel-agents    | ~1,610      |
| receiving-code-review          | ~1,578      |
| using-git-worktrees            | ~1,408      |
| writing-plans                  | ~1,365      |
| using-superpowers              | ~1,304      |
| finishing-a-development-branch | ~1,062      |
| verification-before-completion | ~1,050      |
| requesting-code-review         | ~733        |
| executing-plans                | ~614        |
| **全スキル合計**               | **~29,597** |

→ 全スキルが同時発動すると**約3%**のコンテキストを消費

#### 重要な違い

|                    | Frontend Design | Superpowers                         |
| ------------------ | --------------- | ----------------------------------- |
| スキル数           | 1               | 14                                  |
| 1回の発動で注入    | ~550トークン    | ~600〜5,600トークン（スキルによる） |
| 最大消費量         | ~550トークン    | ~29,597トークン                     |
| コンテキスト占有率 | 0.05%           | 最大約3%                            |

- Superpowersは**同時に全スキルが発動するわけではない**が、1回の作業で複数スキルが連鎖的に発動することはある（例：brainstorming → writing-plans → test-driven-development）
- Frontend Designはスキルが1つだけなので、**常に一定の軽さ**が保証される
- どちらも従来の「ファインチューニング」と違い、コンテキスト内で完結するため**モデル自体は変わらない**
